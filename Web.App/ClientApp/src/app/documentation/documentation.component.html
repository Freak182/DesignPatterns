    Dependency Inversion

    Definition of the Dependency Inversion Principle

    The general idea of this principle is as simple as it is important: High-level modules, which provide complex logic, should be easily reusable and unaffected by changes in low-level modules, which provide utility features. To achieve that, you need to introduce an abstraction that decouples the high-level and low-level modules from each other. Based on this idea, Robert C. Martin's definition of the Dependency Inversion Principle consists of two parts:
    • High-level modules should not depend on low-level modules. Both should depend on abstractions.
    • Abstractions should not depend on details. Details should depend on abstractions.
    An important detail of this definition is, that high-level and low-level modules depend on the abstraction. The design principle does not just change the direction of the dependency, as you might have expected when you read its name for the first time. It splits the dependency between the high-level and low-level modules by introducing an abstraction between them. So in the end, you get two dependencies:
    • the high-level module depends on the abstraction, and
    • the low-level depends on the same abstraction.

    Functional programming

    What is functional programming ?

    According to wikipedia :
    In computer science, functional programming is a programming paradigm where programs are constructed by applying and composing functions. It is a declarative programming paradigm in which function definitions are trees of expressions that map values to other values, rather than a sequence of imperative statements which update the running state of the program.
    Some key concept :
    • You pass function as parameter instead of objects
    • Values are immutable
    • Pure functions - Allows for predictable behavior
     RXJS - What is RXJS ?

    RXJS, c'est le LINQ de JavaScript. Ca permets de programmer des fonctions asynchrone basées sur des événements observables. Les 5 concepts essentiels sont :
    • Observable: represents the idea of an invokable collection of future values or events.
    • Observer: is a collection of callbacks that knows how to listen to values delivered by the Observable.
    • Subscription: represents the execution of an Observable, is primarily useful for cancelling the execution.
    • Operators: are pure functions that enable a functional programming style of dealing with collections with operations like map, filter, concat, reduce, etc.
    • Subject: is equivalent to an EventEmitter, and the only way of multicasting a value or event to multiple Observers.
    • Schedulers: are centralized dispatchers to control concurrency, allowing us to coordinate when computation happens on e.g. setTimeout or requestAnimationFrame or others.
    En front-end, RXJS est souvent considéré comme essentiel a tout désir d'implémentation de programmation fonctionnel.

    SOLID SOLID est un principe de programmation visant a nous encourager a créer des applications plus facilement maintenable, compréhensible et flexible. Ses principes sont :
    • Single Responsibility
    • Open/Closed
    • Liskov Substitution
    • Interface Segregation
    • Dependency Inversion
    Single Responsibility
    Toutes classes ne devraient avoir qu'une seule responsabilité. De plus, elle ne devrait avoir qu'une raison de changer.
    Comment ça nous aide ?
    • Testing - moins de test cases
    • Lower coupling - moins de dépendance
    • Organization - des classes plus petites et mieux organisées
    Open/Closed
    Toutes classes devraient etre ouvertes aux entensions, mais fermé aux modifications. Ainsi, nous nous bloquons de modifier du code existant pouvant causer des régressions.
    Comment ça nous aide ?
    • Réduit les risques de régression
    • Permets de versionner les modifications
    • Réduit les risques de briser des APIs externes ayant une dépendance
    Liskov Substitution
    Si une nouvelle classe remplaçe une existante, a tout moment, nous devrions pouvoir utiliser ou revenir à l'ancienne classe sans rien briser.
    Comment ça nous aide ?
    • Réduit les risques de régression
    • Réduit les risques de briser des APIs externes ayant une dépendance
    Interface Segregation
    Les grandes classes ou interfaçes devraient être découper en plusieurs interfaçes ou classes.
    Comment ça nous aide ?
    • Permets de créer des fonctions ou des classes aux tâches très spécifiques
    • Réduit la complexité des tests à écrires
    Dependency Inversion
    Les classes ou modules devraient pointer vers des abstractions plutôt que d'avoir des dépendances entre-eux.
    Comment ça nous aide ?
    • Permets de créer des fonctions ou des classes aux tâches très spécifiques
    • Réduit la complexité des tests à écrires

    TDD

    What is Test-driven development ?

    According to wikipedia :
    Test-driven development (TDD) is a software development process relying on software requirements being converted to test cases before software is fully developed, and tracking all software development by repeatedly testing the software against all test cases. This is as opposed to software being developed first and test cases created later.
    Some benifits :
    • Reduced Debugging
    • Higher level of documentation
    • Design Improvement
    • Easier refactoring
    Lorsqu'on parle de TDD, nous entendons souvent parler des Three Laws of TDD. Les voici :
    • You are not allowed to write any production code unless it is to make a failing unit test pass
    • You are not allowed to write any more of a unit test than is sufficient to fail; and compilation failures are failures.
    • You are not allowed to write any more production code than is sufficient to pass the one failing unit test.

    Pattern: Saga [Choreography | Orchestration]

    What is the Saga design pattern ?

    C'est une manière de gérer la cohérence au travers un amalgame de microservices lors d'une transaction distribué. C'est une séquence (Saga) de transaction qui met à jour un ensemble de services, chacun émettant un message ou évènement qui trigger la prochaine étape. Nous parlerons parfois d'ochestration ou de chorégraphie.
    *All rights owned by Microsoft

    Quand utiliser ce pattern?

    • Lorsque vous désirez vous assurer de la consistance d'une donnée au travers multiple système
    • Lorsque vous désirez effectuer des rollbacks ou correctifs si un des appels échoue

    Quand ne pas utiliser ce pattern?

    • Transactions fortements couplées
    • Compensating Transaction pattern
    • Dépendances circulaires
    Pattern: Choreography

    What is the Choreography design pattern ?

    C'est une saga avec un point central de coordination. Avec une chorégraphique, chaque transaction émets un événement sur le domaine qui trigger des transactions locales dans les autres services.
    *All rights owned by Microsoft

    Les avantages :

    • Bon pour les séries d'événements simples ayant peu de participants et ne nécessitant pas de logique de coordination
    • Nécessite très peu de maintenance
    • Aucun point d'échec, puisque chaque système est indépendant au travers la saga

    Les désavantages :

    • Complexité d'ajout de nouvels étapes ; Difficulté de tenir comptes de quel service écoute quel événement
    • Complexité des tests d'intégrations
    • Risques de dépendances circulaires

    Pattern: Orchestration

    What is the Orchestration design pattern ?

    L'orchestration est un moyen de coordonner les sagas où un contrôleur centralisé indique aux participants de la saga quelles transactions locales exécuter. L'orchestrateur de la saga gère toutes les transactions et indique aux participants quelle opération effectuer en fonction des événements. L'orchestrateur exécute les requêtes saga, stocke et interprète les états de chaque tâche et gère la reprise après échec avec des transactions compensatoires.
    *All rights owned by Microsoft

    Les avantages :

    • Idéal pour les séries d'événements complexes impliquant de nombreux participants ou de nouveaux participants ajoutés au fil du temps.
    • Approprié quand un contrôle est exercé sur chaque participant au processus et sur la série d'événements
    • N'introduit pas de dépendances cycliques, car l'orchestrateur dépend unilatéralement des participants à la saga.
    • Les participants à la saga n’ont pas besoin de connaître les commandes pour d'autres participants. Une séparation claire des préoccupations simplifie la logique métier.

    Les inconvénients :

    • Une complexité de conception supplémentaire nécessite l'implémentation d'une logique de coordination.
    • Il y a un point de défaillance supplémentaire parce que l'orchestrateur gère le flux de travail complet.

    Migration & DataSeeding

    Migration

    Migration est une méthodologie permettant de faire évoluer votre base de donnée de manière incrémentielle. C'est un outil qui vous permet de faire des back-and-forth sur les différentes versions de votre base de données de manière code-based. Un des gros avantages sera d'être capable de gérer ces migrations à même vos déploiements automatisé.
    Selon Microsoft :
    • Lors de l'introduction d'une modification du modèle de données, le développeur utilise des outils EF Core afin d’ajouter une migration correspondante décrivant les mises à jour nécessaires pour préserver la synchronisation du schéma de la base de données. EF Core compare le modèle actuel à un instantané de l’ancien modèle pour déterminer les différences et génère des fichiers sources de migration. Ces fichiers peuvent, comme n’importe quel fichier source, faire l’objet d’un suivi dans le contrôle de code source du projet.
    • Une fois générée, la nouvelle migration peut être appliquée à une base de données de différentes façons. EF Core enregistre toutes les migrations appliquées dans une table d’historique spéciale, ce qui lui permet de savoir quelles migrations ont été appliquées.

    DataSeed Pattern: How to make migration clean and easy

    Le DataSeed pattern vous permets, une fois migration mis en place, d'interfacer vos migrations et ainsi de les regrouper sous une classe de modification simple. Basée sur le concept de DataSeeding de Microsoft, il redirige plutôt vers une classe donc la seule tâche est de gérer l'ensemble des logiques associées a une nouvelle version d'un object.

    Pattern: Proxy | Api Gateway

    API Gateway Design Pattern

    Selon wikipedia :
    A proxy, in its most general form, is a class functioning as an interface to something else. The proxy could interface to anything: a network connection, a large object in memory, a file, or some other resource that is expensive or impossible to duplicate. In short, a proxy is a wrapper or agent object that is being called by the client to access the real serving object behind the scenes.
    Dans le cadre d'un API, le proxy se manifeste sous forme d'API gateway. Il peut servir de point centrale de gestion des permissions, contrôle du traffic, gestion des micro-frontends et autres. Il sert de point de répartition des communications avec les différents services. On appel souvent ce gateway un BFF lorsque nous parlons d'une application FrontEnd.

    Pattern: State

    State machine

    Le principe d'une state machine permets de suivre l'état d'un object selon une série d'opération.
    Il est généralement composé de :
    1. la classe State, qui définit l'abstraction des comportements du patron
    2. la classe Contexte, qui s'occupe de l'interfaçage du patron avec le reste de son contexte (l'application).
    *All rights owned by SourceMaking
    • La classe State permet via ses spécialisations de créer et gérer dynamiquement les comportements dont l'application aura besoin.
    • La classe Contexte qui permet éventuellement à l'application de choisir le comportement à mettre en œuvre et surtout d'orchestrer l'exécution des comportements.
    Ce pattern peut tant être appliqué à l'état d'un object dans une DataBase, qu'au concept de Template UI. Un example commun sera une page container, gérant la logique d'une certaine transaction, et selon l'état, plusieurs différent template peuvent être appelé.

    Pattern: Mediator

    Mediator pattern in an ASP.NET Web API

    Le modèle médiateur permet de gérer les dépendances entre les objets en les empêchant de communiquer directement entre eux. Au lieu de cela, la communication se fait via un médiateur. Par exemple, un service envoie sa demande au médiateur qui la transmet à son tour au gestionnaire de demandes respectif pour traitement. En ayant un médiateur entre les deux, nous pouvons dissocier les demandes de leurs gestionnaires. L'expéditeur n'a pas besoin de savoir quoi que ce soit sur le gestionnaire. Le modèle médiateur aide à implémenter CQRS. Les commandes et requêtes sont envoyées au médiateur qui les mappe à leurs gestionnaires respectifs.
    *All rights owned by Microsoft

    Les composants :

    • Commands
      • Perform writes, updates, or delete operations on the data
    • Queries
      • Represents data reads
    • Handlers
      • Responsible for receiving the commands, they implement business behavior
